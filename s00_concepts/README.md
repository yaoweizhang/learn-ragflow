## 一、RAG 是什么

### 1.1 LLM 的四大短板

RAG 之所以存在，是因为 LLM 自身有几个绕不开的短板。先从一个日常场景说起：

> 你以前去机场走"二号线换三号线"那条最熟的路线，记住了上回折腾二十多分钟的换乘步骤。这次朋友要去机场，你脱口而出"二号线换三号线"。结果他查了一下，去年新开了机场专线，一站直达——你那个"权威答案"早过时了，越自信越耽误事。

LLM 比你更严重：它的训练语料在训完那一刻就被冻结了，之后的事它统统不知道；它的输出方式又总是流畅自信，所以你很难第一时间发现它在"硬撑"。再加上几个其他问题，LLM 的短板一共可以归纳成四条：

1. **幻觉（Hallucination）** — LLM 本质是基于概率预测下一个 token，并不是"理解"事实。当你问一个看似真实的问题，它会用流畅的语言编造一个听起来很对、其实全错的答案（"一本正经地胡说八道"）。
2. **时效性** — 训练成本极高（一次预训练几百万到几千万美元），训完模型权重就冻结；去年的新闻、再版的法规、刚发布的论文，它统统不知道。
3. **私域知识** — 通用 LLM 只见过公开数据，企业内部文档 / 个人笔记 / 私有合同这类"非公开信息"，对它等于不存在。
4. **上下文窗口限制** — 哪怕你愿意把 10 万字文档全部塞进 prompt，目前主流 LLM 的 context window（4k-128k token）也容不下；超过就要截断，截断就丢信息。

光改 prompt 解决不了这四条；微调（fine-tune）能解私域但成本高且知识更新慢；**最便宜的折中是让模型在回答前去查资料**——这正是 RAG。

### 1.2 RAG 是解药：检索 + 增强 + 生成

**RAG（Retrieval-Augmented Generation，检索增强生成）** 就是上面"让 LLM 答题前去查资料"那条思路的工程化命名。三个动作缺一不可：

- **R - Retrieval（检索）**：从外部知识库里按用户问题查相关文档片段
- **A - Augment（增强）**：把找到的片段拼进 prompt，作为 LLM 的"开卷资料"
- **G - Generation（生成）**：LLM 基于"原问题 + 检索到的资料"作答

三种生活化的比喻帮你记住它：

- **开卷考试**：闭卷 → 开卷，模型从"凭记忆答题"变成"边翻书边答"
- **外挂大脑 / 外挂知识库**：模型本体不变，旁边挂一个可热拔插的资料库
- **可随时翻阅的教科书**：给"会撒谎的优等生"配一本随时可查的工具书，让答案有据可查

> 💡 RAG = 让 LLM 从"闭卷"变"开卷"，给"会撒谎的优等生"配一本可随时翻阅的工具书。

### 1.3 离线 / 在线 两条流水线

RAG 系统的工作流可以拆成两阶段：**离线**那条线把语料变成可检索的索引，**在线**那条线把用户问题变成有据可查的回答。

```
┌─────────────────────────┐                ┌─────────────────────────┐
│  离线：索引构建         │                │  在线：问答             │
│                         │                │                         │
│  文档（PDF/DOCX/        │  ┌──────┐      │  用户问题               │
│  HTML/Markdown）        │─▶│ 切块 │      │       │                 │
│       │                 │  └──────┘      │       ▼                 │
│       ▼                 │       │        │  ┌──────────┐           │
│  ┌──────────┐           │       ▼        │  │ 同款     │           │
│  │ 解析     │           │  ┌─────────┐   │  │ Embedding│           │
│  │ (DeepDoc)│           │  │ Embedding│  │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│       │                 │       │        │       ▼                 │
│       ▼                 │       ▼        │  ┌──────────┐           │
│  ┌──────────┐           │  ┌─────────┐   │  │ 向量召回 │           │
│  │ 结构化   │           │  │ 向量    │   │  │ + BM25   │           │
│  │ (parent) │           │  │ 数据库  │   │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ Rerank   │           │
│                         │                │  └──────────┘           │
│                         │                │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ LLM 生成 │──▶ 回答   │
│                         │                │  └──────────┘           │
└─────────────────────────┘                └─────────────────────────┘
```

- **离线阶段**：文档解析（s02）→ 切块（s03）→ Embedding（s04）→ 写向量库（s05）。
- **在线阶段**：问题 Embedding → 双路召回（向量 + BM25，s06）→ Rerank 精排（s07）→ 拼 Prompt + LLM 生成（s08）。

> "离线"是提前完成的预处理；"在线"是用户发起请求后的实时处理。

### 1.4 三代演进

学术界和工程界普遍把 RAG 演进分成三代（也常叫"三大范式"，但不同来源的标签略有差异——有的写作 "Naive / Advanced / Agentic"，有的写作 "Naive / Advanced / Modular"。本章按后者切，因为后者的 Modular RAG 包含 GraphRAG、Agent 路由等更广义的模块化组合）：

```
  2020-2022 初级   naive RAG: 检索 → 生成(单步)
  ────────────────────────────────────────▶
  2023-2024 高级   advanced RAG: 检索前/检索后/检索中(加 rerank/重写/压缩)
  ────────────────────────────────────────▶
  2024-2025 模块化 modular RAG: 流水线 + Agent 路由(本教程的 s09-s10 走这一支)
  ────────────────────────────────────────▶
        │                  │                     │
        ▼                  ▼                     ▼
     s01/s06              s06-s08               s09/s10
```

| | **初级 RAG（Naive）** | **高级 RAG（Advanced）** | **模块化 RAG（Modular）** |
|---|---|---|---|
| **流程** | 离线：`索引`<br>在线：`检索 → 生成` | 离线：`索引`<br>在线：`… → 检索前 → … → 检索后 → …` | 积木式可编排流程 |
| **特点** | 基础线性流程 | 在检索前后增加优化步骤 | 模块化、可组合、可动态调整 |
| **关键技术** | 基础向量检索 | 查询重写、结果重排（Rerank） | 动态路由、查询转换、多路融合 |
| **局限性** | 效果不稳定、难以优化 | 流程相对固定、优化点有限 | 系统复杂性高 |

## 二、什么时候用 RAG：选型维度

### 2.1 RAG vs 长上下文 vs 微调

在实际工程里，**RAG 不是万能药，也不是首选药**。通用准则：能不改模型就别改模型，成本与可控性都更好。具体选型路径通常是 **提示词工程 → 检索增强生成 → 微调**。

- **横轴 = LLM 优化深度**——从左到右，对模型本身的改动越来越深。提示工程和 RAG **完全不动**模型权重，微调则直接修改参数。
- **纵轴 = 上下文优化深度**——从上到下，给模型的输入信息越来越丰富。提示工程只是改写提问；RAG 通过外挂知识库极大地扩充上下文。

```
           成本 ↓        知识新鲜度 ↑        可控性 ↓
          ───────────────────────────────────────────────
  长上下文  │  ★ (随语料线性增长)   │  ★ (只要塞进 prompt)   │  ★ (无外接依赖)        │
          ───────────────────────────────────────────────
  RAG     │  ★★ (检索一次，长期复用) │  ★★ (索引可热拔插)   │  ★★ (引用 + 拒答可控) │
          ───────────────────────────────────────────────
  微调     │  ★★★ (训练 + 推理双成本)│  ★ (改权重才能更新)   │  ★★★ (行为 / 风格可控)│
          ───────────────────────────────────────────────
```

RAG 在中间，把"低成本 + 高新鲜度 + 中等可控性"这个最难占到的格子拿到手。这也是为什么 RAG 是工业界的事实标准。

如果把维度再展开（是否改动模型 / 成本 / 知识更新 / 优势 / 局限 / 典型场景）：

| 方法 | 是否改动模型 | 成本 | 知识更新 | 优势 | 局限 | 典型场景 |
|---|---|---|---|---|---|---|
| **Prompt 工程** | ❌ 不改 | 低 | 无法更新 | 简单灵活、快速试验 | 效果不稳定、难补知识缺口 | 原型验证、轻量应用 |
| **RAG** | ❌ 不改 | 中 | 替换/追加知识库 | 时效性强、可解释、可引用 | 依赖检索质量、架构复杂 | 企业知识库、科研助手 |
| **Fine-tuning** | ✅ 改 | 高 | 重训模型 | 针对性强、性能上限高 | 成本高、更新慢、有灾难性遗忘 | 法律/医疗/金融垂直模型 |

**经验法则**：轻量任务优先 Prompt 工程；**知识密集型**（要查、要新、要可引用）走 RAG；**高精度定制**（要行为、要风格、要格式）走 Fine-tuning。三者也可叠加——RAG 补知识 + 微调锁行为，是当下最常见的工业组合。

### 2.2 经验法则 + 风险分级

RAG 并非万能。按风险等级给出典型用例与 RAG 适用性：

| 风险等级 | 典型案例 | RAG 适用性 |
|---|---|---|
| **低风险** | 翻译、语法检查 | 高可靠性 |
| **中风险** | 合同起草、法律咨询 | 需结合人工审核 |
| **高风险** | 证据分析、签证决策 | 需严格质量控制机制 |

在 **高风险** 场景里，RAG 自身不足以担责——它的输出必须配合人工复核、规则校验或领域专家评审。

### 2.3 适合谁、不适合谁

- **适合**：客服问答、企业知识库、法规检索、文档型 RAG、检索质量敏感的中等规模场景。
- **不适合**：纯创作 / 闲聊（没有"事实可查"的需求）、高频但语料极小（直接拼 prompt 更省事）、需要"行为 / 风格定制"（应该走微调而非 RAG）。

## 三、本项目与 RAGFlow

### 3.1 RAGFlow 是什么

[RAGFlow](https://github.com/infiniflow/ragflow) 是 GitHub 上 star 数最高的开源 RAG 引擎之一。它把 RAG 全链路的每一环——文档解析、文本切块、embedding、向量索引、混合检索、Rerank、Prompt 模板、Agent 编排、图谱抽取、多模态解析、部署——都做了工程化，写成可独立部署、可独立替换的模块。

RAGFlow 的工程设计原则：**每一环独立可替换；每环可单独生产部署；新接一个 provider（embedding / vector DB / LLM）只需要写一个类 + 在 factory 注册一行**。

### 3.2 为什么选 RAGFlow 做工业参考

- **工程化程度高**——每个模块都有完整接口抽象、生产级实现、单元测试
- **可读性强**——RAGFlow 的代码组织清晰，比 LangChain / LlamaIndex 更适合"做深读"
- **生产实战**——RAGFlow 自带 docker-compose 部署方案，处理过高并发 / 大语料 / 多租户
- **覆盖本教程的每一环**——本教程的 12 章内容 RAGFlow 都实现了工业版

LangChain / LlamaIndex / Dify 等其他框架各有特点（LangChain 抽象层多、LlamaIndex 偏数据框架、Dify 偏可视化），本教程选 RAGFlow 作为**唯一**工业参考，让读者把"30 行玩具 vs 1000 行生产"的对比关系看透。

### 3.3 本教程如何使用 RAGFlow

s02-s12 各章的末尾都有一节 `## RAGFlow 实现`，摘录 RAGFlow 在该层的工程化做法：

- **s01 走全栈骨架**：s01 用 3 段最小代码（子串匹配 / 词袋 / RAG pipeline）跑通 retrieve / augment / generate 的最小闭环；RAGFlow 把这条主干在 s02-s12 拆成 12 个独立模块，每章对一块（见 §1.5）。
- **s02-s12 模块化**：每个 RAGFlow 实现小节含 3 部分——RAGFlow 模块定位（哪个文件/目录）、设计取舍（为什么这样做）、`docs/reference/ragflow-notes/<topic>.md` 链接（5-15 行"为什么这样写"的源码分析）。

**默认阅读路径**：s01-s12 主线 + 工业参考 [`docs/reference/ragflow-notes/`](../docs/reference/ragflow-notes/) 摘录并行推进。

## 思考题答案

### Q1. 为什么"参数化 + 非参数化按需结合"比单独说"检索 / 增强 / 生成"中任一词都重要？

任何一项单独都不能解决 LLM 的真实问题：

- 单说"检索"——LLM 拿到一堆 chunk 也不会自动答对（不会"增强"生成）
- 单说"增强"——没有外部检索来源，"增强"的是模型自己的参数化记忆，跟闭卷答题没区别
- 单说"生成"——这是所有 LLM 都能做的，谈不上 RAG 的价值

只有三者按需结合——按问题类型决定"主要靠参数化记忆"还是"主要靠外部检索"、再让 LLM 把两类知识综合成答案——RAG 才成立。这跟"开卷考试"是一个比喻：检索 = 翻书、增强 = 把书摊在桌上、生成 = 写出答案。三个动作缺一不可。

### Q2. RAG vs 长上下文 vs 微调，三者的取舍矩阵里 RAG 卡在哪个格子？

RAG 卡在"低成本 + 高新鲜度 + 中等可控性"这个格子：
- **成本**：检索一次建索引，长期复用——比长上下文（每次都塞全语料）便宜得多
- **新鲜度**：索引可热拔插——比微调（改权重才能更新）灵活得多
- **可控性**：引用 [i] + 拒答 prompt 模板——介于长上下文（无外接依赖）和微调（行为完全可控）之间

这个三角的"左下角"是工业界最常卡的位置：高频 + 大语料 + 重复问题几乎总是 RAG 便宜。

### Q3. 三代 RAG 的"分水岭"是"加了 Rerank"还是"加了 Agent 路由"？为什么？

**加了 Agent 路由**才是真正的分水岭。

- 初级 → 高级：Rerank / 查询重写 / HyDE / 多路召回 都是**流程内的优化**——还是一条直线，只是直线上每个节点更精细。流程结构没变。
- 高级 → 模块化：Agent 路由让流程从**直线变成图**——同一个 query 可能走"向量召回 → LLM"、也可能走"BM25 → 重写 → 向量召回 → LLM"、还可能走"工具调用 → 多跳 → 融合"。流程不再固定，按 query 动态选路。这是从"优化既有流程"到"动态编排流程"的范式跃迁。

所以分水岭在 Agent 路由，Rerank 只是高级 RAG 内部的关键技术之一（s07 详谈），模块化 RAG 的关键技术在 s09（Agent）+ s10（GraphRAG）。

### Q4. 本教程 12 章中，s06（混合检索）和 s07（重排序）属于哪一代？s09（Agent）和 s10（GraphRAG）呢？

- s06（混合检索）——基础链路是初级 RAG；但 s06 加上 α-weighted BM25+dense 融合属于**高级 RAG**（在检索环节做了优化）
- s07（重排序）——典型**高级 RAG** 特征（在检索后做精排）
- s09（Agent）——典型**模块化 RAG** 特征（动态路由 + 工具调用）
- s10（GraphRAG）——**高级 RAG 的延伸**（多跳检索），但实现上走模块化管线（按 query 类型走不同检索路径），所以也归到模块化 RAG
