### 1.1 核心定义：什么是 RAG

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种把"外部知识检索"与"大语言模型生成"组合起来的技术范式。它的核心思想是在 LLM 生成回答之前，先从外部知识库里**按问题查相关文档**，把找到的内容作为上下文拼进 prompt，再让 LLM 基于这段上下文作答。

```
   用户问题 ──▶ 检索 ──▶ 上下文片段 ──▶ 增强 prompt ──▶ LLM ──▶ 答案
                │              │             │             │
                ▼              ▼             ▼             ▼
              向量/BM25      chunk[1..k]   "基于以下资料回答"  生成(token级)
   (非参数化)            (非参数化)       (联结器)         (参数化)
```

> 💡 **一句话总结**：RAG = 检索（Retrieval）+ 增强（Augmented）+ 生成（Generation）。
>
> 让 LLM 学会**开卷考试**——既能利用模型自己学到的"参数化知识"，也能随时查阅外部"非参数化知识"。

所谓**参数化知识**，是 LLM 在训练阶段从海量文本里学到的、固化在数十亿参数权重里的"模糊记忆"；所谓**非参数化知识**，是存在外部知识库里、可随时增删改的"精准资料"。RAG 把这两类知识**按需结合**——这一结合本身，比 RAG 这个名词里任何单独一个字都更重要。

RAG 系统的工作流可以拆成两阶段，**离线**那条线把语料变成可检索的索引，**在线**那条线把用户问题变成有据可查的回答：

```
┌─────────────────────────┐                ┌─────────────────────────┐
│  离线：索引构建         │                │  在线：问答             │
│                         │                │                         │
│  文档（PDF/DOCX/        │  ┌──────┐      │  用户问题               │
│  HTML/Markdown）        │─▶│ 切块 │      │       │                 │
│       │                 │  └──────┘      │       ▼                 │
│       ▼                 │       │        │  ┌──────────┐           │
│  ┌──────────┐           │       ▼        │  │ 同款     │           │
│  │ 解析     │           │  ┌─────────┐   │  │ Embedding│           │
│  │ (DeepDoc)│           │  │ Embedding│  │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│       │                 │       │        │       ▼                 │
│       ▼                 │       ▼        │  ┌──────────┐           │
│  ┌──────────┐           │  ┌─────────┐   │  │ 向量召回 │           │
│  │ 结构化   │           │  │ 向量    │   │  │ + BM25   │           │
│  │ (parent) │           │  │ 数据库  │   │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ Rerank   │           │
│                         │                │  └──────────┘           │
│                         │                │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ LLM 生成 │──▶ 回答   │
│                         │                │  └──────────┘           │
└─────────────────────────┘                └─────────────────────────┘
```

- **离线阶段**：文档解析（s02）→ 切块（s03）→ Embedding（s04）→ 写向量库（s05）。
- **在线阶段**：问题 Embedding → 双路召回（向量 + BM25，s06）→ Rerank 精排（s07）→ 拼 Prompt + LLM 生成（s08）。

> 💡 "离线"是提前完成的预处理；"在线"是用户发起请求后的实时处理。

### 1.2 三代演进 + 12 章映射

学术界普遍把 RAG 演进分成三代，下面的对比表把每代的核心特征并列出来。

```
  2020-2022 初级   naive RAG: 检索 → 生成(单步)
  ────────────────────────────────────────▶
  2023-2024 高级   advanced RAG: 检索前/检索后/检索中(加 rerank/重写/压缩)
  ───────────────────────────────────────▶
  2024-2025 模块化 modular RAG: 流水线 + Agent 路由(本教程的 s09-s10 走这一支)
  ────────────────────────────────────────▶
        │                  │                     │
        ▼                  ▼                     ▼
     s01/s06              s06-s08               s09/s10
```

| | **初级 RAG（Naive RAG）** | **高级 RAG（Advanced RAG）** | **模块化 RAG（Modular RAG）** |
|---|---|---|---|
| **流程** | 离线：`索引`<br>在线：`检索 → 生成` | 离线：`索引`<br>在线：`… → 检索前 → … → 检索后 → …` | 积木式可编排流程 |
| **特点** | 基础线性流程 | 在检索前后增加优化步骤 | 模块化、可组合、可动态调整 |
| **关键技术** | 基础向量检索 | 查询重写、结果重排（Rerank） | 动态路由、查询转换、多路融合 |
| **局限性** | 效果不稳定、难以优化 | 流程相对固定、优化点有限 | 系统复杂性高 |

本教程的章节对应关系：s01-s08 是**初级 RAG** 的最小骨架；s06 + s07 进入**高级 RAG**（混合检索 + Rerank）；s09（Agent）+ s10（GraphRAG）+ s12（部署）属于**模块化 RAG** 的不同切片。

### 1.3 选型对比：RAG vs 长上下文 vs 微调

在实际工程里，**RAG 不是万能药，也不是首选药**。通用准则：能不改模型就别改模型，成本与可控性都更好。具体选型路径通常是 **提示词工程 → 检索增强生成 → 微调**。

- **横轴 = LLM 优化深度**——从左到右，对模型本身的改动越来越深。提示工程和 RAG **完全不动**模型权重，微调则直接修改参数。
- **纵轴 = 上下文优化深度**——从下到上，给模型的输入信息越来越丰富。提示工程只是改写提问；RAG 通过外挂知识库极大地扩充上下文。

```
           成本 ↓        知识新鲜度 ↑        可控性 ↓
          ───────────────────────────────────────────────
  长上下文  │  ★ (随语料线性增长)   │  ★ (只要塞进 prompt)   │  ★ (无外接依赖)        │
          ───────────────────────────────────────────────
  RAG     │  ★★ (检索一次，长期复用) │  ★★ (索引可热拔插)   │  ★★ (引用 + 拒答可控) │
          ───────────────────────────────────────────────
  微调     │  ★★★ (训练 + 推理双成本)│  ★ (改权重才能更新)   │  ★★★ (行为 / 风格可控)│
          ───────────────────────────────────────────────
```

RAG 在中间，把"低成本 + 高新鲜度 + 中等可控性"这个最难占到的格子拿到手。这也是为什么 RAG 是工业界的事实标准。

RAG 解决了 LLM 的四类典型局限：

| LLM 的局限 | RAG 的解法 |
|---|---|
| **静态知识**：训练截止后的事 | 实时检索外部知识库，支持动态更新 |
| **幻觉（Hallucination）** | 基于检索内容生成，错误率显著降低 |
| **领域专业性不足** | 引入领域特定知识库（医疗、法律、企业内部） |
| **数据隐私风险** | 本地化部署知识库，敏感数据不出域 |

### 1.4 适合谁、不适合谁

- **适合**：客服问答、企业知识库、法规检索、文档型 RAG、检索质量敏感的中等规模场景。
- **不适合**：纯创作 / 闲聊（没有"事实可查"的需求）、高频但语料极小（直接拼 prompt 更省事）、需要"行为 / 风格定制"（应该走微调而非 RAG）。

## 二、RAG 是什么：参数化 vs 非参数化：[code_01_what_is_rag.py](code_01_what_is_rag.py)

### 这是什么

最核心的一句话：RAG = 检索 + 增强 + 生成。把 LLM 从"闭卷"变成"开卷"——既能利用模型自己学到的"参数化知识"，也能随时查阅外部"非参数化知识"。两个知识按需结合，比 RAG 这个名词里任何单独一个字都更重要。

入口：[`code_01_what_is_rag.py`](code_01_what_is_rag.py)

### 跑起来

```bash
python s00_concepts/code_01_what_is_rag.py
```

本脚本用 Python 标准库演示"参数化 vs 非参数化"的概念差——不调 LLM，不依赖任何外部服务。输出会展示一段事实性陈述在"完全靠模型记忆" vs "外挂知识库"两种模式下答案的稳定性差异。

### 它做对了什么

- **零依赖、零成本**：纯 stdlib，5 秒跑完。

- **没区分初级 / 高级 / 模块化**——三代演进的区别见 03。

## 三、为什么用 RAG：vs 长上下文 vs 微调：[code_02_why_rag.py](code_02_why_rag.py)

### 这是什么

一张"选型二维图 + 风险分级表"的浓缩版：**横轴是 LLM 优化深度、纵轴是上下文优化深度**——Prompt / RAG / 微调 在这张图上有清晰的分界；RAG 不是万能药，也不总是首选药。这一节解决"RAG 是不是适合我手头这个任务"的判断问题。

入口：[`code_02_why_rag.py`](code_02_why_rag.py)

### 跑起来

```bash
python s00_concepts/code_02_why_rag.py
```

本脚本用 stdlib 算一笔小账：把"长上下文塞进 prompt"和"RAG 检索 + 拼上下文"两种方案，按 token 数 × 单价折算成美元并排打印出来。结论一眼可见——**高频 + 大语料 + 重复问题** 几乎总是 RAG 便宜。

### 它做对了什么

- **零依赖**：纯 stdlib，单文件 5 秒跑完。
- **把"何时用 RAG"量化**：token × 单价 × QPS 三件套变成具体数字，不再凭感觉。

### 它做错了什么（这就是后面章节要解决的）

- **没考虑检索质量成本**——RAG 便宜的前提是"检索召回了对的资料"；召错了一样会幻觉。检索质量的成本在 s06-s07 详谈。
- **没考虑索引维护成本**——"索引热拔插"听起来美好，但向量库需要持续写入、监控、去重，这部分成本没算进去；s05 / s12 详谈。
- **风险分级表是经验值**——具体业务的风险等级需要法务 / 业务方共同判定；本表只起提示作用。

```
Q: R3630 G5 配备多少内存插槽?
A: (LLM draws on parametric memory only)

[with RAG]
<context>

[2] (server_whitepaper.pdf#4) 五、可靠性 内存支持镜像、备用与纠错码(ECC)三种数据保护模式。

A: (LLM grounded by non-parametric context)

差异:同一个问题,带 RAG 时 prompt 里多了 <context> 块,
LLM 可以'引用' [1][2] 给答案背书 — 这就是增强(Augmented)。
```

**02 跑出来（实测，零依赖；`tiktoken` 没装时退化为 char/4）：**

```
策略     每次 query 喂给 LLM 的 token 数(粗估)
--------------------------------------------------
长上下文     400050 tokens  (全语料每次都塞,贵)
RAG            1250 tokens  (只塞 top-3 命中,便宜 320x)
微调            13 tokens  (知识在权重里,但改不动)

代价矩阵:
- 长上下文: 改知识秒级,但 token 钱按 query × 语料烧
- RAG:      改知识秒级,且只塞相关段,钱按 query × top-k 烧
- 微调:     改知识按小时/天,改完后查询最便宜,但失去了'热拔插'

经验法则:能不动模型就别动模型(RAG > 微调);
知识实时性高 + 体积大 → RAG;模型行为/风格定制 → 微调。
```

```
  离线环节数: 4    在线环节数: 3
```

### 适用场景的风险分级

RAG 并非万能。下表按风险等级给出典型用例与 RAG 适用性：

| 风险等级 | 典型案例 | RAG 适用性 |
|---|---|---|
| **低风险** | 翻译、语法检查 | 高可靠性 |
| **中风险** | 合同起草、法律咨询 | 需结合人工审核 |
| **高风险** | 证据分析、签证决策 | 需严格质量控制机制 |

在 **高风险** 场景里，RAG 自身不足以担责——它的输出必须配合人工复核、规则校验或领域专家评审。这也是为什么**"可溯源"特性在严肃场景比在闲聊场景重要得多**。

## 思考题答案

### 01 / Q1. 为什么"参数化 + 非参数化按需结合"比单独说"检索 / 增强 / 生成"中任一词都重要？

任何一项单独都不能解决 LLM 的真实问题：

- 单说"检索"——LLM 拿到一堆 chunk 也不会自动答对（不会"增强"生成）；
- 单说"增强"——没有外部检索来源，"增强"的是模型自己的参数化记忆，跟闭卷答题没区别；
- 单说"生成"——这是所有 LLM 都能做的，谈不上 RAG 的价值。

只有三者按需结合——按问题类型决定"主要靠参数化记忆"还是"主要靠外部检索"、再让 LLM 把两类知识综合成答案——RAG 才成立。这跟"开卷考试"是一个比喻：检索 = 翻书、增强 = 把书摊在桌上、生成 = 写出答案。三个动作缺一不可，缺一个要么是闭卷、要么是抄书、要么是写作文。

### 01 / Q2. 在线阶段哪一步是不可省的？哪一步是可选的（属于"高级 RAG"）？

不可省：**问题 Embedding → 召回 → LLM 生成**。少任何一步整条链路就断了——不 embed 召回不到东西，不召回 prompt 里没上下文，不生成 LLM 不出文本。

可选：**Rerank 精排**（s07）、**查询重写 / HyDE**（s07）、**多路召回 + 融合**（s06）、**Agent 路由**（s09）——这些是"高级 RAG"和"模块化 RAG"的优化项，把检索质量 / 召回率 / 适应性往上抬，但 MVP 不加它们也能跑（参考 s01 03 的最小链路）。

### 01 / Q3. 如果让你给同事 5 分钟讲清楚 RAG，这张 ASCII 图你会怎么讲？

三句话：

1. **RAG = 开卷考试**：LLM 本来是闭卷答题（只靠训练时学的"参数化知识"），RAG 给它外挂一份"参考资料"（"非参数化知识"），每次答题先查资料再写答案。
2. **离线 / 在线两条线**：离线那条线提前把资料整理成可查的索引（解析 → 切块 → embed → 写库）；在线那条线每次只查相关段（问题 embed → 召回 → LLM）。整条 RAG 链路就是这两条线的接口。
3. **比单说"检索"或"生成"都重要**：检索 ≠ 增强 ≠ 生成，三者必须按需结合——按问题类型决定靠模型记忆还是靠外部资料，再用 LLM 综合成答案。

### 02 / Q1. 一个 1000 QPS、语料 1 TB 的客服问答系统，Prompt / RAG / 微调 哪个最划算？为什么？

**RAG**。粗算一下：

- 1 TB 文本 ≈ 250 亿 token（按 4 char/token）。长上下文方案：每次 query 塞 250 亿 token × 1000 QPS = 25 万亿 token/s；按 $1 / 1M tokens 算，单 token 成本就烧光预算。
- 微调方案：单 query token 成本最低（只需 question），但 1 TB 语料要喂进权重——按 1 TB × 1 epoch 算 ≈ 250 亿 token 训练，单次成本极高；且客服知识每天都在变（产品上新、政策调整），每次更新都要重新训练。
- RAG 方案：单 query 只塞 top-3 命中 ≈ 1000 token × 1000 QPS = 100 万 token/s；语料更新只需重建索引（分钟级），模型不动。token 成本 ≈ 长上下文的 1/250000，知识更新成本 ≈ 微调的 1/N（N 是训练 epoch 数）。

客服问答是"高频 + 大语料 + 重复问题"的典型组合——RAG 在这张三角的左下角（成本 + 新鲜度 + 可控性）刚好卡在最优格子。

### 02 / Q2. RAG 解决了 LLM 的四类局限，对哪一类它其实只是"缓解"而不是"根治"？

**幻觉（Hallucination）**。其他三类（静态知识 / 领域专业性 / 数据隐私）都可以靠"换一份好资料"根治：

- 静态知识 → 索引里加新数据就行；
- 领域专业性 → 索引里塞领域资料就行；
- 数据隐私 → 知识库本地部署就行。

但幻觉不是"知识不够"造成的，是 LLM 在压力下会"顺着 prompt 编故事"——RAG 给它参考资料后，**有据可查的部分**大幅减少幻觉，但**模型对资料的"理解 + 总结 + 拼接"环节**仍可能编。比如检索到 3 段互相矛盾的 chunk，模型可能在拼接时把互相冲突的事实"和稀泥"成一句错话。**根治幻觉要的是**：① Rerank 精排（s07）让 top-k 尽量一致；② Prompt 工程强制拒答/引用（s08）；③ 评估工具（RAGAS / TruLens）端到端把关——RAG 是第一道防线，不是唯一防线。

### 02 / Q3. 索引热拔插听起来美好，但实操里哪些场景会出问题？

**冷启动一致性**：刚换完索引的瞬间，旧索引还在被一些 in-flight 请求读取，新索引还没被所有实例加载——会有"答非所问"窗口。生产里通常靠"双写 → 灰度 → 全量切"三步走。

**版本回滚**：发现新索引有 bug 想切回旧版，旧索引如果已经被 GC 掉了就回不去。生产里通常保留 N 个历史版本（一般 3-5 份），方便 5 分钟内回滚。

**跨实例缓存**：向量库实例往往有 query cache / embedding cache，热拔插不刷缓存就会"明明索引已更新但召回还是旧的"。生产里通常在切换时主动 `cache.clear()` 或重启服务。

**审计追溯**：法务 / 业务方问"这个答案是哪份索引返回的"——需要把"query → 答案 → 索引版本 ID"留痕。生产里通常在 metadata 里打 `index_version` + `index_build_ts` 字段。

**权限矩阵**：多租户场景下不同租户的索引隔离没做对，热拔插可能让租户 A 误读到租户 B 的资料。生产里通常靠 namespace / collection 隔离 + 切换前的 dry-run。

这些问题不致命但生产一定要考虑——s12 部署章会再回头谈。

### 03 / Q1. 三代 RAG 的"分水岭"是"加了 Rerank"还是"加了 Agent 路由"？为什么？

**加了 Agent 路由**才是真正的分水岭。

- 初级 RAG → 高级 RAG：Rerank / 查询重写 / HyDE / 多路召回 都是**流程内的优化**——还是一条直线，只是直线上每个节点更精细。Rerank 让 top-k 更准、查询重写让 query 更易召回，但**流程结构没变**。
- 高级 RAG → 模块化 RAG：Agent 路由让流程从**直线变成图**——同一个 query 可能走"向量召回 → LLM"、也可能走"BM25 → 重写 → 向量召回 → LLM"、还可能走"工具调用 → 多跳 → 融合"。流程不再固定，按 query 动态选路。这是从"优化既有流程"到"动态编排流程"的范式跃迁。

所以分水岭在 Agent 路由，Rerank 只是高级 RAG 内部的关键技术之一（s07 详谈），模块化 RAG 的关键技术在 s09（Agent）+ s10（GraphRAG）。

### 03 / Q2. 给你一段生产 RAG 系统的需求描述，怎么判断它属于哪一代？该往哪个方向演？

按三个维度问：

1. **流程是固定一条线还是会变？** 固定 → 初级或高级；按 query / 上下文动态变 → 模块化。
2. **有没有"重排 / 重写 / 压缩"这类检索前后优化？** 没有 → 初级；有 → 高级。
3. **有没有"工具调用 / 多跳 / 多模态"这类旁路？** 没有 → 初级或高级；有 → 模块化。

举例：

- "公司内部 FAQ 检索，回答模板固定" → 初级（s01-s06 即可上线）。
- "公司内部 FAQ + 检索结果要按部门权限过滤 + 高频问题走 FAQ、低频走 RAG" → 高级（s06 + s07 + Prompt 工程控制拒答）。
- "复杂合同审查，要从多份 PDF 抽取条款 + 跨条款关联 + 必要时调外部 API 查法规" → 模块化（s09 Agent + s10 GraphRAG + 工具调用）。

演进的建议：**先 MVP（初级）上线 → 用 Rerank / 混合检索进入高级 → 用 Agent 路由进入模块化**。三段不是替代关系，是叠加关系。

### 03 / Q3. 模块化 RAG 的"灵活性"和"系统复杂性"怎么平衡？有没有"过度模块化"的反例？

**反例：把所有可能都用 Agent 编排**——一个内部 FAQ 系统，明明 5 步直线能答完，非要 Agent 路由 + 工具调用 + 多路召回，结果：

1. 调试成本指数级上升——每条 query 的路径不固定，问题复现困难；
2. 评估难度陡增——同一类 query 可能走不同路径，A/B 测试失去对照；
3. 引入新的失败模式——Agent 解析失败、工具调用超时、循环调用……MVP 阶段根本不需要承担这些成本；
4. 性能损耗——Agent 决策 + 多轮 LLM 调用把 latency 从 1 秒拉到 5-10 秒。

**平衡原则**：

- **MVP 阶段**：直线流程（初级 → 高级），不上 Agent；
- **遇到"按 query 走不同路径"的真实需求**（比如有些 query 走 FAQ、有些走 RAG、有些走工具调用）：**针对性**地加 Agent 路由，不要全场景上；
- **多跳 / 多模态 / 跨系统**等不可避免的复杂场景才上模块化；
- **每加一层抽象就加一个评估点**——RAGAS / TruLens 的端到端指标是"灵活性有没有换来效果"的唯一判官。

过度模块化的本质是"为了灵活性牺牲了可观测性 + 可调试性"，这两个特性在生产里比灵活性更重要。s09 / s12 会再回头谈这个权衡。
