## 一、什么是 RAG？

### 1.1 核心定义

```
   用户问题 ──▶ 检索 ──▶ 上下文片段 ──▶ 增强 prompt ──▶ LLM ──▶ 答案
                │              │             │             │
                ▼              ▼             ▼             ▼
              向量/BM25      chunk[1..k]   "基于以下资料回答"  生成(token级)
   (非参数化)            (非参数化)       (联结器)         (参数化)
```

> **图 1-1**　RAG 的 4 个环节：检索增强 + LLM 生成的流水线。左边两段是非参数化知识，右边一段是参数化知识，中间的 prompt 是"联结器"。

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种把"外部知识检索"与"大语言模型生成"组合起来的技术范式。它的核心思想是在 LLM 生成回答之前，先从外部知识库里**按问题查相关文档**，把找到的内容作为上下文拼进 prompt，再让 LLM 基于这段上下文作答。

> 💡 **一句话总结**：RAG = 检索（Retrieval）+ 增强（Augmented）+ 生成（Generation）。
>
> 让 LLM 学会**开卷考试**——既能利用模型自己学到的"参数化知识"，也能随时查阅外部"非参数化知识"。

所谓**参数化知识**，是 LLM 在训练阶段从海量文本里学到的、固化在数十亿参数权重里的"模糊记忆"；所谓**非参数化知识**，是存在外部知识库里、可随时增删改的"精准资料"。RAG 把这两类知识**按需结合**——这一结合本身，比 RAG 这个名词里任何单独一个字都更重要。

### 1.2 技术原理：检索 + 生成两阶段

RAG 系统的工作流可以拆成两阶段，**离线**那条线把语料变成可检索的索引，**在线**那条线把用户问题变成有据可查的回答：

```
┌─────────────────────────┐                ┌─────────────────────────┐
│  离线：索引构建         │                │  在线：问答             │
│                         │                │                         │
│  文档（PDF/DOCX/        │  ┌──────┐      │  用户问题               │
│  HTML/Markdown）        │─▶│ 切块 │      │       │                 │
│       │                 │  └──────┘      │       ▼                 │
│       ▼                 │       │        │  ┌──────────┐           │
│  ┌──────────┐           │       ▼        │  │ 同款     │           │
│  │ 解析     │           │  ┌─────────┐   │  │ Embedding│           │
│  │ (DeepDoc)│           │  │ Embedding│  │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│       │                 │       │        │       ▼                 │
│       ▼                 │       ▼        │  ┌──────────┐           │
│  ┌──────────┐           │  ┌─────────┐   │  │ 向量召回 │           │
│  │ 结构化   │           │  │ 向量    │   │  │ + BM25   │           │
│  │ (parent) │           │  │ 数据库  │   │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ Rerank   │           │
│                         │                │  └──────────┘           │
│                         │                │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ LLM 生成 │──▶ 回答   │
│                         │                │  └──────────┘           │
└─────────────────────────┘                └─────────────────────────┘
```

- **离线阶段**：文档解析（s02）→ 切块（s03）→ Embedding（s04）→ 写向量库（s05）。
- **在线阶段**：问题 Embedding → 双路召回（向量 + BM25，s06）→ Rerank 精排（s07）→ 拼 Prompt + LLM 生成（s08）。

> 💡 "离线"是提前完成的预处理；"在线"是用户发起请求后的实时处理。

> RAG 的三代演进（初级 / 高级 / 模块化）放到 [unit 03](../03_evolution/) 单独讲——这一节只锚定"什么 = 检索 + 增强 + 生成"。

## 这是什么

最核心的一句话：RAG = 检索 + 增强 + 生成。把 LLM 从"闭卷"变成"开卷"——既能利用模型自己学到的"参数化知识"，也能随时查阅外部"非参数化知识"。两个知识按需结合，比 RAG 这个名词里任何单独一个字都更重要。

## 跑起来

```bash
python s00_concepts/units/01_what_is_rag/code.py
```

本 unit 用 Python 标准库演示"参数化 vs 非参数化"的概念差——不调 LLM，不依赖任何外部服务。输出会展示一段事实性陈述在"完全靠模型记忆" vs "外挂知识库"两种模式下答案的稳定性差异。

## 它做对了什么

- **零依赖、零成本**：纯 stdlib，5 秒跑完。
- **把"开卷 vs 闭卷"数字化**：让"参数化 vs 非参数化"这两个抽象名词变成可对比的具体输出。
- **后续 12 章的锚点**：s01-s12 的每一段代码，最终都对应这张 ASCII 图里的一条边。

## 它做错了什么（这就是后面章节要解决的）

- **没真做检索**——只演示概念骨架；真正能从外部知识库查东西的代码见 s01-s05。
- **没真接 LLM**——为了保持零依赖，生成那一步用 mock 模拟；真接 LLM 的最小链路见 s01 unit 3。
- **没区分初级 / 高级 / 模块化**——三代演进的区别见 [unit 03](../03_evolution/)。
