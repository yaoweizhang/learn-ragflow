## 一、什么是 RAG？

### 1.1 核心定义

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种把"外部知识检索"与"大语言模型生成"组合起来的技术范式。它的核心思想是在 LLM 生成回答之前，先从外部知识库里**按问题查相关文档**，把找到的内容作为上下文拼进 prompt，再让 LLM 基于这段上下文作答。

> 💡 **一句话总结**：RAG = 检索（Retrieval）+ 增强（Augmented）+ 生成（Generation）。
>
> 让 LLM 学会**开卷考试**——既能利用模型自己学到的"参数化知识"，也能随时查阅外部"非参数化知识"。

所谓**参数化知识**，是 LLM 在训练阶段从海量文本里学到的、固化在数十亿参数权重里的"模糊记忆"；所谓**非参数化知识**，是存在外部知识库里、可随时增删改的"精准资料"。RAG 把这两类知识**按需结合**——这一结合本身，比 RAG 这个名词里任何单独一个字都更重要。

### 1.2 技术原理：检索 + 生成两阶段

RAG 系统的工作流可以拆成两阶段，**离线**那条线把语料变成可检索的索引，**在线**那条线把用户问题变成有据可查的回答：

```
┌─────────────────────────┐                ┌─────────────────────────┐
│  离线：索引构建         │                │  在线：问答             │
│                         │                │                         │
│  文档（PDF/DOCX/        │  ┌──────┐      │  用户问题               │
│  HTML/Markdown）        │─▶│ 切块 │      │       │                 │
│       │                 │  └──────┘      │       ▼                 │
│       ▼                 │       │        │  ┌──────────┐           │
│  ┌──────────┐           │       ▼        │  │ 同款     │           │
│  │ 解析     │           │  ┌─────────┐   │  │ Embedding│           │
│  │ (DeepDoc)│           │  │ Embedding│  │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│       │                 │       │        │       ▼                 │
│       ▼                 │       ▼        │  ┌──────────┐           │
│  ┌──────────┐           │  ┌─────────┐   │  │ 向量召回 │           │
│  │ 结构化   │           │  │ 向量    │   │  │ + BM25   │           │
│  │ (parent) │           │  │ 数据库  │   │  └──────────┘           │
│  └──────────┘           │  └─────────┘   │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ Rerank   │           │
│                         │                │  └──────────┘           │
│                         │                │       │                 │
│                         │                │       ▼                 │
│                         │                │  ┌──────────┐           │
│                         │                │  │ LLM 生成 │──▶ 回答   │
│                         │                │  └──────────┘           │
└─────────────────────────┘                └─────────────────────────┘
```

- **离线阶段**：文档解析（s02）→ 切块（s03）→ Embedding（s04）→ 写向量库（s05）。
- **在线阶段**：问题 Embedding → 双路召回（向量 + BM25，s06）→ Rerank 精排（s07）→ 拼 Prompt + LLM 生成（s08）。

> 💡 "离线"是提前完成的预处理；"在线"是用户发起请求后的实时处理。

### 1.3 技术演进：三代 RAG

学术界普遍把 RAG 演进分成三代 [^1][^2]，下面的对比表 1-1 把每代的核心特征并列出来：

<div align="center">

| | **初级 RAG（Naive RAG）** | **高级 RAG（Advanced RAG）** | **模块化 RAG（Modular RAG）** |
|---|---|---|---|
| **流程** | 离线：`索引`<br>在线：`检索 → 生成` | 离线：`索引`<br>在线：`… → 检索前 → … → 检索后 → …` | 积木式可编排流程 |
| **特点** | 基础线性流程 | 在检索前后增加优化步骤 | 模块化、可组合、可动态调整 |
| **关键技术** | 基础向量检索 | 查询重写、结果重排（Rerank） | 动态路由、查询转换、多路融合 |
| **局限性** | 效果不稳定、难以优化 | 流程相对固定、优化点有限 | 系统复杂性高 |

## 二、为什么要用 RAG？

### 2.1 技术选型：Prompt → RAG → 微调

在实际工程里，**RAG 不是万能药，也不是首选药**。通用准则：能不改模型就别改模型，成本与可控性都更好。具体选型路径通常是 **提示词工程 → 检索增强生成 → 微调**。

我们可以从一个二维图来理解这三者的位置：

- **横轴 = LLM 优化深度**——从左到右，对模型本身的改动越来越深。提示工程和 RAG **完全不动**模型权重，微调则直接修改参数。
- **纵轴 = 上下文优化深度**——从下到上，给模型的输入信息越来越丰富。提示工程只是改写提问；RAG 通过外挂知识库极大地扩充上下文。

基于这个二维图，三者的分工就清晰了：

- **先尝试提示工程**——任务简单、模型已有相关知识时，精心设计的 prompt 就够用。
- **再选择 RAG**——当模型缺乏特定或实时知识而答不上来时，用 RAG 给它外挂一份"参考资料"。
- **最后考虑微调**——只有当目标是**改变模型"如何做"**（行为、风格、格式），而不是**"知道什么"**（事实性知识）时，微调才是合适的。例如让模型严格遵循某种独特的输出格式、模仿特定人物的对话风格，或把极其复杂的指令"蒸馏"进权重。

RAG 解决了 LLM 的四类典型局限：

| LLM 的局限 | RAG 的解法 |
|---|---|
| **静态知识**：训练截止后的事 | 实时检索外部知识库，支持动态更新 |
| **幻觉（Hallucination）** | 基于检索内容生成，错误率显著降低 |
| **领域专业性不足** | 引入领域特定知识库（医疗、法律、企业内部） |
| **数据隐私风险** | 本地化部署知识库，敏感数据不出域 |

### 2.2 关键优势

（1）**准确性与可信度的双重提升**

RAG 最核心的价值是突破了模型预训练知识的边界。它不仅能**补充专业领域的知识盲区**，还能通过提供具体参考材料有效**抑制"一本正经胡说八道"的幻觉**。研究还表明，RAG 生成内容在**具体性**和**多样性**上显著优于纯 LLM [^3]。更重要的是 **可溯源性**——每一条回答都能找到对应的原始文档出处，这种"有据可查"的特性在法律、医疗等严肃场景下尤其关键。

（2）**时效性保障**

RAG 解决了 LLM 的**知识时滞**——模型不知道训练截止日期之后发生的事。RAG 允许知识库**独立于模型动态更新**：新政策或新数据一旦入库，立刻就能被检索到。这种能力被社区称为 **"索引热拔插"（Index Hot-swapping）**——就像给机器人换一张存储卡，瞬间切换其世界知识库，无需重新训练模型。

（3）**显著的综合成本效益**

RAG 是一种高性价比的方案。首先，它**避免了高频微调**带来的巨额算力成本；其次，有了外部知识的辅助，我们往往可以用**参数量更小的基础模型**达到类似效果，直接降低推理成本；最后，它减少了"把海量知识强行塞进模型权重"的资源消耗。

（4）**灵活的模块化可扩展性**

RAG 架构天然支持**多源集成**（PDF、Word、网页、数据库），且**检索与生成解耦**——可以独立优化检索组件（比如换更好的 Embedding 模型），而不会影响生成组件的稳定性。

### 2.3 适用场景的风险分级

RAG 并非万能。下表 1-2 按风险等级给出典型用例与 RAG 适用性：

| 风险等级 | 典型案例 | RAG 适用性 |
|---|---|---|
| **低风险** | 翻译、语法检查 | 高可靠性 |
| **中风险** | 合同起草、法律咨询 | 需结合人工审核 |
| **高风险** | 证据分析、签证决策 | 需严格质量控制机制 |

<p><em>表 1-2 RAG 适用场景风险分级</em></p>

在 **高风险** 场景里，RAG 自身不足以担责——它的输出必须配合人工复核、规则校验或领域专家评审。这也是为什么**"可溯源"特性在严肃场景比在闲聊场景重要得多**。

## 三、如何上手 RAG？

### 3.1 工具链选择

构建一个 RAG 系统涉及几层选型：

- **开发模式**：可用 **LangChain / LlamaIndex** 快速集成，也可**不依赖框架**写原生代码——后者在 AI 编程辅助下并不难，且能让你看清每一步发生什么（本教程走的就是原生路线）。
- **向量数据库**：大规模生产可选 **Milvus / Pinecone / Weaviate**；轻量本地可选 **FAISS / Chroma**；本教程用 Chroma 做演示（s05）。
- **Embedding 模型**：中文场景常用 **BGE 系列**、**M3E**；英文可用 **OpenAI text-embedding-3**；多语言可用 **Cohere embed-multilingual-v3**。
- **评估工具**：上线前用 **RAGAS** 或 **TruLens** 做端到端评估（检索相关性 + 生成质量）。

### 3.2 四步构建最小可行系统（MVP）

如果你要在两天内搭一个能用的 RAG 演示，按以下四步走：

1. **数据准备与清洗**——把 PDF、Word、网页等多源数据解析成统一文本流。注意表格、扫描件、公式这些特殊元素（s02 / s11 详解）。分块策略优先按**语义段落**切，避免固定字符数把句子切碎（s03 详解）。
2. **索引构建**——用 Embedding 模型把文本块转成向量，写入向量数据库。这一步关联**元数据**（来源、页码、章节），对后续精确引用非常关键（s04 / s05）。
3. **检索策略优化**——不要依赖单一的向量搜索。采用**混合检索**（向量 + BM25）提升召回率，再引入**重排序**模型对 top-k 做精排（s06 / s07）。
4. **生成与提示工程**——设计清晰的 Prompt 模板，把 top-N hits 渲染成 `[i] (source#page) text` 形式拼进上下文；明确要求模型"不知道就说不知道"，并提供引用格式以便 UI 端渲染（s08）。

> 这四步正好对应本教程的 s02 → s05 → s06/s07 → s08。你可以把每一章当作 MVP 在该步骤上的最小骨架。

### 3.3 进阶与挑战

基础的 MVP 搭起来后，下一步的关键是**评估、诊断、突破瓶颈**。

**评估维度**：

- **检索相关性**：top-k 中是否包含答案所在段落？
- **生成质量**：进一步拆分为**语义准确性**（意思对不对）和**词汇匹配度**（专业术语用没用对）。
- **可溯源性**：回答中的每一条事实能否回溯到具体文档？

**典型挑战**：

- **检索依赖性**——如果召回了错误信息，再强的 LLM 也会"一本正经胡说八道"。
- **多跳推理**——需要跨多段文档综合分析的问题，常见架构普遍吃力；可考虑 GraphRAG（s10）。
- **长上下文冲突**——当 top-k 里有相互矛盾的资料时，模型往往随机选边，需要 Prompt 工程兜底。

**优化方向**：

- **性能层**：索引分层（对高频数据启用缓存）、多模态扩展（支持图像/表格检索，s11）。
- **架构层**：从线性流程演进到**设计模式**——**分支模式**并行处理多路检索、**循环模式**进行自我修正（s09 ReAct 是典型例子）。

## 四、RAG 已死？

随着大模型长上下文窗口能力的提升，社区里开始出现"RAG 已死"的声音。这种论调主要来自两个方面：一是认为长上下文已经能暴力"消化"海量文本，不再需要复杂检索；二是批评 RAG 这个术语本身就过于宽泛，模糊了太多技术细节。

这些观点忽略了一个概念在演进过程中的普遍规律。我们可以为现代复杂的 RAG 系统起一个更精确、更唬人的名字，比如 **"大模型知识管理专家系统"（Large Language Model Knowledge Management Expert System, LKE）**。它由三个核心要素组成：

- **L（Large Language Model）**：系统的驱动力是大语言模型。
- **K（Knowledge Management）**：系统就像知识管理员，精准地**检索**到所需知识，辅助后续更高阶应用。
- **E（Expert）**：系统能像专家一样，通过路由、分析、融合、修正一系列步骤，最终**生成**答案、解决问题。

但这种"换名游戏"恰恰说明了"RAG 已死"论的表面化——无异于用一个新瓶子去装 RAG 这个不断陈化的老酒。

可以类比 **Transformer**。今天无论是以 GPT 为代表的 Decoder-only 还是以 BERT 为代表的 Encoder-only，我们都习惯称之为"基于 Transformer 架构"，尽管它们与原始论文中的完整形态差异巨大。Transformer 这个标签抓住了一次技术范式的核心飞跃，并成为了一个时代的象征。

同理，**RAG 的核心是"将 LLM 的内在参数化知识与外部非参数化知识相结合"**。只要这个思想不变，无论我们为其增加多少模块——查询转换、多路召回、自我修正——它本质上依然是在这个框架下的演进。

所以，**"RAG 已死"是一个伪命题**。相反，RAG 作为一个概念活得很好——它正在像 Transformer 一样，成为一个不断吸收新技术、不断进化的基础架构范式。它的生命力，正在于它的"面目全非"和"包罗万象"。

> 本教程的目标，是绘制出 RAG 全貌的清晰地图。当你能够解构它的每一个模块、理解它的每一种可能性时，RAG 也好，LKE 也罢，这些标签都无关紧要。我们要做的，是通过 RAG 这道经典例题，掌握"将 LLM 的内在参数化知识与外部非参数化知识相结合"这一类题型的解题思路。
