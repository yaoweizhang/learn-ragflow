# RAG 技术综述：什么是 RAG，为什么，怎么做

## 1. RAG 是什么

### 1.1 核心定义

**RAG（Retrieval-Augmented Generation）** 是一种把"外部知识检索"与"大语言模型生成"组合起来的技术范式。它的核心思路是：在 LLM 生成回答之前，先从外部知识库里**按问题查相关文档**，把找到的内容作为上下文拼进 prompt，再让 LLM 基于这段上下文作答。

> **一句话总结**：RAG = 检索（Retrieval）+ 增强（Augmented）+ 生成（Generation）。  
> "让 LLM 学会开卷考试"——既能利用模型自己学到的参数化知识，也能随时查阅外部资料。

### 1.2 为什么需要 RAG：LLM 的三种失败

LLM 看似什么都知道，但有三类问题它答不好：

1. **训练截止之后的事**——"我们公司去年营收多少"、"今天上海天气"。模型只见过截止日之前的语料，对你的私有数据、当下动态一无所知。
2. **私有 / 内部数据**——公司年报、内部 wiki、合同 PDF。这些从未在公开互联网上出现过，模型自然没学过。
3. **幻觉（Hallucination）**——当模型不知道答案时，它常常一本正经地瞎编，尤其是数字、人名、引文这类"看起来很确定"的内容。LLM 默认是"无据可查"。

RAG 直接针对这三点：

- **可更新**——知识换文档就行，不用重训模型；
- **可私有**——把内部文档放进知识库，模型就能引用；
- **可溯源**——回答里可以贴"第几条资料支撑了我这句"，幻觉显著减少。

### 1.3 RAG 不是把 LLM 训练一遍

容易混淆：RAG **不是**对 LLM 做继续训练或微调，而是把"查资料"这个动作外置。这有几层意义：

- **成本**：训练一个 7B 模型要几十到几百 GPU 小时；RAG 的瓶颈在**索引的构建成本**（一次性）+ **检索延迟**（每请求），通常低一个数量级。
- **可控性**：训练完的模型行为改不动，要改只能 retrain。RAG 改文档、改正文、改 prompt 都能立即生效。
- **可解释**：训练的黑盒只能解释输入→输出；RAG 能告诉你"这条答案是从文档 X 第 Y 段召回的"。

## 2. RAG 工作流

RAG 系统通常分**离线索引**和**在线问答**两条线：

```
┌─────────────────────┐                ┌─────────────────────┐
│  离线：索引构建     │                │  在线：问答         │
│                     │                │                     │
│  文档（PDF/DOCX/   │  ┌──────┐      │  用户问题          │
│  HTML/Markdown）   │─▶│ 切块 │      │       │             │
│       │             │  └──────┘      │       ▼             │
│       ▼             │       │        │  ┌──────────┐       │
│  ┌──────────┐       │       ▼        │  │ 同款    │       │
│  │ 解析     │       │  ┌─────────┐   │  │ Embedding│       │
│  │ (DeepDoc)│       │  │ Embedding│  │  └──────────┘       │
│  └──────────┘       │  └─────────┘   │       │             │
│       │             │       │        │       ▼             │
│       ▼             │       ▼        │  ┌──────────┐       │
│  ┌──────────┐       │  ┌─────────┐   │  │ 向量召回 │       │
│  │ 结构化   │       │  │ 向量    │   │  │ + BM25   │       │
│  │ (parent) │       │  │ 数据库  │   │  └──────────┘       │
│  └──────────┘       │  └─────────┘   │       │             │
│                     │                │       ▼             │
│                     │                │  ┌──────────┐       │
│                     │                │  │ Rerank   │       │
│                     │                │  └──────────┘       │
│                     │                │       │             │
│                     │                │       ▼             │
│                     │                │  ┌──────────┐       │
│                     │                │  │ LLM 生成 │──▶ 回答│
│                     │                │  └──────────┘       │
└─────────────────────┘                └─────────────────────┘
```

### 2.1 离线：索引构建

1. **文档解析**——从 PDF/DOCX/HTML/Markdown 等多源异构格式抽成结构化文本。注意 PDF 里的表格、扫描件、公式这些特殊元素（后面 s11 详解）。
2. **结构化**——把所有段落按"页面 / 标题 / 段落 / 表格 / 图"等粒度标好类型。
3. **切块（Chunking）**——把长文切成"检索单位"的小块。RAG 系统最怕的就是"输入即垃圾"，切块策略直接决定召回质量（s03 详解）。
4. **Embedding**——把每个 chunk 映射到一个高维向量（BGE / OpenAI / Cohere 等模型）。
5. **向量数据库**——存 chunk + 向量 + 元数据，支持按相似度召回 + 按 metadata 过滤。

### 2.2 在线：问答

1. **用户提问** → 同款 embedding → 向量；
2. **召回**——向量近邻（如 Chroma / FAISS / Milvus）+ 关键词匹配（BM25 / Elasticsearch）双路召回；
3. **加权融合**——两路分数归一化，按 `α * 向量 + (1-α) * BM25` 加权；
4. **精排（Rerank）**——cross-encoder 重排序（BGE reranker / Cohere / Jina）；
5. **拼 prompt**——把 top-N hits 渲染成 `[i] (source#page) text`，拼进 prompt；
6. **LLM 生成**——基于"只能依据 `<context>` 内的资料"约束生成回答，必要时拒答（"I don't know"）；
7. **角标 + 引用溯源**——把 LLM 写的 `[1]` 解析回 source/page，还给上游 / UI。

### 2.3 完整链路的关键转折点

每一环都有可能让 RAG 失效。从工程经验来看，**最贵的不是 LLM 调用**，而是：

- **文档解析**——扫描件/表格/公式抽不出来，下游全空；
- **切块策略**——长 paragraph 把无关内容塞进 hit，LLM 被噪声带偏；
- **Embedding 模型**——中英混排时 multilingual 模型选错，召回全乱；
- **Rerank**——双塔 bi-encoder 对精确匹配不敏感，rerank 没接好，召回 top-1 都不对；
- **Prompt 模板**——"<context>" 的边界没界定好，prompt injection 一打就穿。

## 3. 技术演进：Naive → Advanced → Modular

学术界普遍把 RAG 演进分成三代：

### 3.1 Naive RAG（初级 RAG）

最朴素的"离线索引 → 在线 检索 → 生成"。

- **优点**：实现简单，几十行代码就够。
- **缺点**：检索召回差（错位匹配、吞上下文）、生成慢（一段一查）、无引用溯源、无拒答。
- **本教程 s01-s08** 就是这一代：把每个组件最小化跑通。

### 3.2 Advanced RAG（高级 RAG）

在 Naive 基础上加 **检索前 / 检索后**的优化：

- **检索前**：query rewriting（改写问题）、query expansion（拓展同义词）、HyDE（用 LLM 生成假设答案再查）；
- **检索后**：rerank、re-rank prompt、context compression（只把相关句喂给 LLM）；
- **索引侧**：hierarchical indexing、multi-vector indexing（一个 document 多个视角）；
- **本教程 s06 + s07** 就是这一代：α 加权融合 + cross-encoder 精排。

### 3.3 Modular RAG（模块化 RAG）

更进一步的解耦，把整条管线拆成可组合的模块（query router / retriever / reranker / generator / verifier），允许**动态选择**走哪条路径。

- **代表性框架**：RAGFlow（agent/canvas.py 的 Canvas DAG）、LangGraph、LlamaIndex 的 `RouterQueryEngine`。
- **本教程 s09（ReAct 循环）+ s10（GraphRAG）** 涉及 Agent/工具路由，s12（FastAPI 包装）是部署层的 Modular。

## 4. RAG vs. 长上下文 vs. 微调

实际工程里你经常要在三选一中决策：

| 维度 | RAG | 长上下文（百万 token） | 微调 |
|---|---|---|---|
| **数据新鲜度** | 立刻可换文档 | 仍受 cutoff 限制 | 需重训 |
| **私有数据** | 任意大小 | 受上下文窗口限制 | 需训练数据 + 算力 |
| **可解释性** | 高（可贴引用） | 低 | 低 |
| **延迟** | 中（先检索再生成） | 直接生成 | 直接生成 |
| **成本** | 文档索引一次性 | token 单价高 | GPU 小时 |
| **适用场景** | 知识库问答 / 客服 / 文档助手 | 单文档摘要 / 多文档综合 | 风格 / 行为 / 推理链定制 |

**经验法则**：

- 你**有大量垂直知识**（公司文档、行业报告）→ 先 RAG，再考虑混合；
- 你**想改的是模型行为**（口吻、风格、指令遵从）→ 微调；
- 你**单文档超长**（大部头财报 / 整本 PDF）→ 长上下文（注意 token 成本）；
- 你**三者都有需要**→ 不用选，组合用：RAG 喂事实 + 微调喂风格 + 长上下文给当前文档。

## 5. RAG 何时帮倒忙

不是所有场景都该上 RAG：

- **数据规模太小**（< 1KB）→ 不需要 RAG，直接塞 prompt 就够了。
- **任务本身不需要事实**（写诗、改写、风格迁移）→ RAG 引入噪声，没收益。
- **数据高度结构化**（SQL 强查询）→ GraphRAG / Text2SQL 更合适（s10 涉及 GraphRAG）。
- **延迟要求极严**（< 100ms）→ RAG 检索+生成通常 1-3s，不合适。
